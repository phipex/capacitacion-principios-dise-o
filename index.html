Principios Fundamentales de Diseño de Software

Una guía para construir software robusto, mantenible y escalable. Aprende a escribir código que no solo funciona, sino que perdura en el tiempo.

1. Legibilidad

El software debe ser comprensible. Un código legible es más fácil de depurar, mantener y mejorar.

Unicidad

Cada concepto o pieza de información debe tener una única representación en el sistema, evitando la duplicidad de código.

Relación Principal: Implementación directa de DRY (Don't Repeat Yourself).

Qué hacer:

  • ✓Centralizar la lógica de negocio.
  • ✓Crear funciones de utilidad compartidas.
  • ✓Usar constantes o enumeraciones.

Qué no hacer:

  • ✗Copiar y pegar bloques de código.
  • ✗Definir la misma lógica en múltiples capas.
  • ✗Repetir consultas idénticas.

Preguntas que debo hacerme:

  • Si necesito cambiar esta lógica, ¿en cuántos lugares debo hacerlo?
  • ¿Esta pieza de código ya existe en algún otro lugar del proyecto?
  • ¿Estoy escribiendo algo que sé que ya he escrito antes?

Características del software:

  • Mantenimiento rápido: Un cambio se hace en un solo lugar.
  • Menos propenso a errores: Se elimina el riesgo de inconsistencias.
  • Código más limpio: La base de código es más pequeña y fácil de navegar.

Regla Práctica:

"Una vez, dos veces, tres veces, refactoriza."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Un sistema donde la validación de un email se realiza de tres formas distintas: una vez en el controlador de la API (usando una regex diferente), otra vez en el servicio (con una función de validación duplicada) y una tercera vez en el frontend (con una lógica JavaScript repetida).

Cómo cumplirlo

Una clase `EmailValidator` centralizada con un método `isValid(email)` que encapsula toda la lógica de validación del email. Esta clase es inyectada o utilizada por el controlador de la API, el servicio y el código del frontend.

Uniformidad

Se debe aplicar un conjunto coherente y mínimo de criterios en todo el software, como en la nomenclatura de variables y funciones.

Relación Principal: Pilar de KISS (Keep It Simple, Stupid). Un código uniforme es más simple de mantener.

Qué hacer:

  • ✓Definir y automatizar guías de estilo.
  • ✓Usar los mismos patrones de diseño.
  • ✓Mantener una estructura de carpetas consistente.

Qué no hacer:

  • ✗Permitir que cada desarrollador aplique su propio estilo.
  • ✗Mezclar patrones sin una razón clara.
  • ✗Usar diferentes formas de nombrar los mismos conceptos.

Preguntas que debo hacerme:

  • ¿Un nuevo desarrollador podría entender la estructura rápidamente?
  • ¿La forma en que estoy nombrando es consistente con el resto?
  • ¿Estamos resolviendo este tipo de problema de la misma manera en otros lugares?

Características del software:

  • Menor carga cognitiva: El código es predecible y fácil de "escanear".
  • Onboarding más rápido: Los nuevos miembros se vuelven productivos antes.
  • Revisiones de código eficientes: El equipo se enfoca en la lógica y no en el estilo.

Regla Práctica:

"El estilo de código es como la señalización vial: debe ser consistente para que todos entiendan."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Un archivo JavaScript donde algunas variables se nombran con `camelCase`, otras con `snake_case` y algunas constantes con minúsculas. Las llaves de apertura de bloques `if` o `for` están a veces en la misma línea y a veces en una nueva línea.

Cómo cumplirlo

Un proyecto configurado con un linter (ej. ESLint) que aplica automáticamente reglas de estilo (ej. `camelCase` para variables, `PascalCase` para clases, identación con 2 espacios). Todas las funciones y clases siguen la misma convención.

Minimización

El software debe tener únicamente la funcionalidad necesaria, evitando agregar características que no se van a utilizar.

Relación Principal: Idéntico a YAGNI (You Ain't Gonna Need It).

Qué hacer:

  • ✓Implementar únicamente la funcionalidad descrita.
  • ✓Escribir la solución más simple y directa.
  • ✓Refactorizar solo cuando un nuevo requisito lo justifique.

Qué no hacer:

  • ✗Añadir funcionalidad "por si acaso".
  • ✗Sobre-ingenieriar soluciones.
  • ✗Construir por especulación o por una necesidad irreal.

Preguntas que debo hacerme:

  • ¿Hay un requisito explícito que pida esto ahora mismo?
  • ¿Cuál es la forma más simple en la que esto podría funcionar?
  • ¿Estoy construyendo esto por especulación o por una necesidad real?

Características del software:

  • Entrega rápida de valor: El equipo se enfoca en lo que importa ahora.
  • Mayor flexibilidad: Es más fácil adaptar un código simple que uno complejo.
  • Menos código que mantener: Se reduce la superficie para la aparición de bugs.

Regla Práctica:

"Implementa lo que sabes que necesitas hoy, no lo que crees que necesitarás mañana."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Un módulo de gestión de usuarios que incluye funciones para exportar datos a formatos XML, CSV y JSON, aunque el único requisito actual es la exportación a CSV. Además, tiene una función `importarUsuariosDesdeExcel` que nunca se ha solicitado.

Cómo cumplirlo

El módulo de gestión de usuarios solo contiene la función `exportarUsuariosACsv`. Si en el futuro surge un requisito para exportar a JSON, se añadiría esa funcionalidad en ese momento, siguiendo la misma estructura y patrón.

2. Modularidad

Construir el software a partir de piezas interconectadas pero independientes. La modularidad mejora la mantenibilidad, la reutilización y la capacidad de trabajar en paralelo.

Abstracción

Exponer interfaces simples que ocultan la complejidad interna. El consumidor de un módulo no necesita saber *cómo* funciona, solo *qué* hace.

Relación Principal: La ocultación (o encapsulación) es un pilar de la POO y es fundamental para el principio Information Expert de GRASP.

Qué hacer:

  • ✓Exponer solo lo necesario a través de interfaces públicas.
  • ✓Ocultar la complejidad interna y los datos (privado).
  • ✓Ofrecer métodos que representen "qué" hacen, no "cómo".

Qué no hacer:

  • ✗Exponer todos los miembros de una clase como públicos.
  • ✗Crear clases solo con getters y setters (Anemic Domain Model).
  • ✗Permitir que el código cliente manipule el estado interno.

Preguntas que debo hacerme:

  • ¿Realmente necesito que este método o propiedad sea público?
  • ¿Estoy exponiendo detalles de implementación que el cliente no necesita saber?

Características del software:

  • Seguridad: Se protege el estado interno de los objetos.
  • Mantenibilidad: Se puede cambiar la implementación interna sin afectar a quienes lo consumen.
  • Simplicidad: Los consumidores interactúan con una interfaz simple.

Regla Práctica:

"Oculta el 'cómo', expón el 'qué'."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Una clase `Coche` donde para cambiar la velocidad, el código cliente tiene que acceder directamente a `coche.motor.cilindros.setRpm(X)` y `coche.transmision.engranajeActual = Y`, exponiendo detalles de cómo funciona el motor y la transmisión.

Cómo cumplirlo

La clase `Coche` expone un método público simple `acelerar(incrementoVelocidad)`. La lógica interna de cómo se ajustan el motor y la transmisión para lograr esa velocidad está encapsulada dentro del método, sin que el cliente necesite conocer los detalles.

Ocultación

Los módulos deben ocultar los detalles de implementación que no son esenciales para su interacción con otros módulos.

Relación Principal: Pilar de POO y fundamental para Information Expert de GRASP.

Qué hacer:

  • ✓Exponer solo lo necesario vía interfaces públicas.
  • ✓Ocultar la complejidad interna (privado).
  • ✓Ofrecer métodos que representen "qué" hacen, no "cómo".

Qué no hacer:

  • ✗Exponer todos los miembros como públicos.
  • ✗Crear clases solo con getters y setters (Anemic Domain Model).
  • ✗Permitir que el cliente manipule el estado interno.

Características del software:

  • Seguridad: Se protege el estado interno de los objetos.
  • Mantenibilidad: Se puede cambiar la implementación interna sin afectar a los consumidores.
  • Simplicidad: Los consumidores interactúan con una interfaz simple.

Regla Práctica:

"Por defecto, todo es privado. Hazlo público solo si es estrictamente necesario."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Una clase `CarritoCompras` donde todos los atributos y métodos auxiliares son públicos, permitiendo que cualquier parte del código externo pueda manipular directamente el estado interno del carrito.

Cómo cumplirlo

La clase `CarritoCompras` declara sus atributos y métodos auxiliares como privados. Solo expone métodos públicos esenciales como `addItem(producto)`, `removeItem(producto)`, y `getTotal()`.

Autoprotección

Los módulos deben incluir validaciones y condiciones para protegerse contra un uso inadecuado.

Relación Principal: Se relaciona con Diseño por Contrato y asegura la robustez (KISS).

Qué hacer:

  • ✓Validar argumentos de métodos públicos (guard clauses).
  • ✓Asegurar que los objetos se creen en estado válido.
  • ✓Lanzar excepciones claras ante uso incorrecto.

Qué no hacer:

  • ✗Confiar ciegamente en los datos de entrada.
  • ✗Permitir la creación de objetos en estados inválidos.
  • ✗Capturar excepciones de manera silenciosa.

Preguntas que debo hacerme:

  • ¿Qué pasaría si a este método le llega un valor nulo o inesperado?
  • ¿Está este objeto protegiendo su propio estado y reglas (invariantes)?

Características del software:

  • Robustez: Más resistente a fallos por datos inesperados.
  • Depuración fácil: Los errores se detectan cerca de la fuente del problema.
  • Seguridad: Se evitan estados inconsistentes que pueden llevar a vulnerabilidades.

Regla Práctica:

"No confíes en el código que te llama; valida tus entradas."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Una función `dividir(a, b)` que no verifica si `b` es cero, lo que lleva a un error de división por cero en tiempo de ejecución si se le pasa `0` como segundo argumento.

Cómo cumplirlo

La función `dividir(a, b)` incluye una `guard clause` al inicio: `if (b === 0) { throw new Error("No se puede dividir por cero."); }`. Esto asegura que el método solo opere con entradas válidas.

Cohesión

Las partes que componen un módulo deben estar estrechamente relacionadas y enfocadas en una única responsabilidad.

Relación Principal: Corresponde a Alta Cohesión (GRASP) y Responsabilidad Única (SRP) de SOLID.

Qué hacer:

  • ✓Agrupar datos y funciones que operan sobre ellos.
  • ✓Asegurar que un módulo tenga una única responsabilidad.

Qué no hacer:

  • ✗Crear clases genéricas como `Utils` o `Helpers`.
  • ✗Mezclar lógica de negocio, acceso a datos y UI en una clase.

Preguntas que debo hacerme:

  • ¿Puedo describir lo que hace esta clase en una oración sin usar la palabra "y"?
  • Si cambio la UI, ¿debería modificar esta clase de lógica de negocio? (Si es sí, la cohesión es baja).

Características del software:

  • Fácil de entender: Los módulos tienen un propósito claro.
  • Reutilizable: Los módulos enfocados son más fáciles de usar en otros contextos.
  • Robusto: El impacto de un cambio está localizado.

Regla Práctica:

"Si una clase hace más de una cosa, divídela."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Una clase `GestorDeReportes` que contiene métodos para generar informes de ventas (lógica de negocio), conectarse a la base de datos (acceso a datos), enviar emails con los informes (comunicación externa) y formatear fechas (utilidad).

Cómo cumplirlo

Se tienen clases separadas: `ReporteVentasService` (lógica), `EmailSender` (envío), `ReporteRepository` (acceso a datos) y `DateFormatter` (utilidad). `ReporteVentasService` utiliza las otras para cumplir su única responsabilidad.

Desacoplamiento

Se debe minimizar el número de dependencias entre módulos.

Relación Principal: Se alinea con Bajo Acoplamiento (GRASP) y es el objetivo de DIP e ISP de SOLID.

Qué hacer:

  • ✓Depender de interfaces o abstracciones.
  • ✓Utilizar inyección de dependencias.
  • ✓Comunicarse entre módulos a través de eventos.

Qué no hacer:

  • ✗Instanciar dependencias concretas directamente.
  • ✗Que los módulos se llamen entre sí de forma cruzada.
  • ✗Que un cambio en una clase obligue a corregir otras 10.

Preguntas que debo hacerme:

  • ¿Puedo reemplazar este componente sin modificar las clases que lo usan?
  • ¿Puedo probar esta clase de forma aislada?

Características del software:

  • Flexibilidad: Permite cambiar o reemplazar partes con un impacto mínimo.
  • Testeabilidad: Los componentes se pueden probar de forma aislada.
  • Paralelización del trabajo: Equipos distintos pueden trabajar sin interferir entre sí.

Regla Práctica:

"Depende de abstracciones, no de concreciones."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Una clase `OrderProcessor` que, dentro de su método, crea directamente una instancia concreta de `new PayPalPaymentGateway()` y `new EmailNotificationService()`. Está fuertemente acoplado a estas implementaciones.

Cómo cumplirlo

La clase `OrderProcessor` recibe una `IPaymentGateway` y una `INotificationService` como dependencias en su constructor. Las implementaciones concretas se inyectan desde fuera, permitiendo cambiar de proveedor sin modificar `OrderProcessor`.

Granularidad

Los módulos deben ser pequeños en todas sus dimensiones: pocas líneas de código, pocos atributos y pocos métodos.

Relación Principal: Apoya a SRP, Alta Cohesión e ISP.

Qué hacer:

  • ✓Crear clases y métodos pequeños y enfocados.
  • ✓Dividir interfaces grandes en varias más pequeñas.
  • ✓Limitar el número de parámetros en los métodos.

Qué no hacer:

  • ✗Crear "clases dios" (God Classes).
  • ✗Definir métodos con una larga lista de parámetros.
  • ✗Forzar a una clase a implementar métodos que no necesita.

Preguntas que debo hacerme:

  • ¿Este método hace más de una cosa?
  • ¿Esta clase tiene demasiadas dependencias o responsabilidades?
  • ¿Estoy forzando a los clientes a depender de cosas que no usan?

Características del software:

  • Comprensibilidad: Es más fácil entender un componente pequeño.
  • Reutilización: Los componentes pequeños son perfectos para ser reutilizados.
  • Testeabilidad: Es mucho más sencillo escribir pruebas para una unidad pequeña.

Regla Práctica:

"Las funciones deben hacer una cosa y hacerla bien; las clases, una responsabilidad."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Un método `procesarPedidoCompleto` con 200 líneas de código que maneja la validación del usuario, el cálculo del precio, la persistencia en la base de datos, el envío de notificaciones y la actualización del inventario.

Cómo cumplirlo

El método `procesarPedido` es mucho más pequeño y delega las tareas a métodos o clases más pequeños y específicos: `validarPedido()`, `calcularPrecio()`, `guardarPedidoEnBD()`, `enviarNotificacion()`, `actualizarInventario()`.

Jerarquía

La estructura del software debe ser jerárquica, evitando las dependencias circulares entre módulos.

Relación Principal: Se alinea con el Principio de Inversión de Dependencias (DIP).

Qué hacer:

  • ✓Organizar dependencias para que fluyan en una sola dirección.
  • ✓Utilizar DIP para romper ciclos.

Qué no hacer:

  • ✗Permitir dependencias circulares (A -> B y B -> A).
  • ✗Que una capa de bajo nivel dependa de una de alto nivel.

Preguntas que debo hacerme:

  • Si dibujo un grafo de las dependencias, ¿hay ciclos?
  • ¿Las dependencias fluyen de lo abstracto a lo concreto o viceversa?

Características del software:

  • Comprensible: La estructura del proyecto es clara.
  • Mantenible: Evita el "efecto dominó" donde un cambio rompe todo.

Regla Práctica:

"Las flechas de dependencia siempre deben apuntar hacia las abstracciones o capas de mayor nivel."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Un proyecto donde la capa de "presentación" depende de la de "dominio", la de "dominio" de la de "infraestructura", pero la de "infraestructura" también depende de la de "dominio" (un ciclo).

Cómo cumplirlo

El proyecto sigue una arquitectura de capas (cebolla o hexagonal). Las dependencias fluyen hacia el centro (dominio). La capa de infraestructura depende de puertos (interfaces) definidos en el dominio, no al revés.

Dominio/Negocio y Tecnología

Es fundamental separar las partes del software que representan la lógica del negocio de las que se encargan de los aspectos tecnológicos.

Relación Principal: Base de arquitecturas como la Hexagonal o Limpia (Clean Architecture).

Qué hacer:

  • ✓Crear un núcleo de negocio sin dependencias de frameworks.
  • ✓Tratar la tecnología como un "plugin" o detalle.

Qué no hacer:

  • ✗Mezclar código de un framework en la lógica de negocio.
  • ✗Que el modelo de dominio dependa de la estructura de la BD.

Preguntas que debo hacerme:

  • Si cambio de framework, ¿tengo que reescribir mi lógica de negocio?
  • ¿Puedo probar mis reglas de negocio sin una base de datos?

Características del software:

  • Longevo: La lógica de negocio sobrevive a los cambios tecnológicos.
  • Adaptable: Es fácil cambiar de proveedor de BD, framework, etc.
  • Altamente testeable: El núcleo se puede probar sin dependencias externas.

Regla Práctica:

"Tu lógica de negocio debe ser agnóstica a la tecnología que la soporta."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Una entidad `Producto` en el dominio que tiene anotaciones específicas de un framework ORM (`@Entity`, `@Table`) mezcladas con la lógica de negocio.

Cómo cumplirlo

La entidad `Producto` es una clase POJO sin anotaciones. Existe una interfaz `IProductoRepository` en el dominio, y su implementación `JpaProductoRepository` (que usa anotaciones) reside en la capa de infraestructura.

3. Extensibilidad

Facilitar la adición de nuevo código sin modificar el existente. Un diseño extensible permite que el software evolucione de manera controlada y segura.

Generalización

Los módulos deben ser responsables de las características más generales de un problema.

Relación Principal: Es la definición del Principio Abierto/Cerrado (OCP) de SOLID.

Qué hacer:

  • ✓Identificar comportamiento común y colocarlo en una clase base.
  • ✓Implementar variaciones en clases derivadas.
  • ✓Usar Polimorfismo (GRASP).

Qué no hacer:

  • ✗Crear jerarquías de herencia muy profundas.
  • ✗Poner lógica específica de una subclase en la clase base.

Preguntas que debo hacerme:

  • ¿Cuál es el concepto general aquí y cuáles son las variaciones específicas?
  • ¿Estoy usando herencia para reutilizar código o para representar una relación "es un/a"?

Características del software:

  • Reutilización de código: El comportamiento general se escribe una sola vez.
  • Claridad conceptual: El código refleja la estructura del dominio del problema.
  • Extensible: Es fácil añadir nuevas especializaciones sin tocar el código general.

Regla Práctica:

"Identifica el patrón común; encapsula la variación."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Un conjunto de clases `Perro`, `Gato`, `Pájaro`, cada una con su propio método `dormir()`, `comer()`, `hacerSonido()`, replicando mucha lógica idéntica o muy similar en cada una, sin una clase base `Animal`.

Cómo cumplirlo

Existe una clase abstracta `Animal` con métodos concretos `dormir()` y `comer()` que encapsulan la lógica común. Luego, `Perro`, `Gato` y `Pájaro` extienden `Animal` y solo implementan su comportamiento específico para `hacerSonido()`.

Especialización

Cada módulo derivado de uno general debe representar una única alternativa o caso de uso específico.

Relación Principal: Mecanismo para implementar OCP y LSP, y se relaciona con Polimorfismo (GRASP).

Qué hacer:

  • ✓Crear subclases que implementen comportamientos específicos.
  • ✓Sobrescribir métodos abstractos o virtuales.
  • ✓Añadir funcionalidades específicas de la especialización.

Qué no hacer:

  • ✗Añadir lógica genérica en la especialización.
  • ✗Romper el contrato de la clase base.
  • ✗Modificar la clase base para acomodar nuevas especializaciones.

Preguntas que debo hacerme:

  • ¿Esta especialización representa una única variante del concepto general?
  • ¿Puedo describir el propósito de esta especialización de forma concisa?

Características del software:

  • Modularidad mejorada: Cada especialización es un módulo independiente.
  • Reutilización del diseño: Se reutilizan las abstracciones generales.
  • Fácil adición de nuevas variantes: Añadir especializaciones es sencillo.
  • Código claro y fácil de entender: Cada parte tiene un propósito claro.

Regla Práctica:

"Cada derivación debe ser una única variación específica, sin sorpresas."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Una clase `Cuadrado` que hereda de `Rectangulo`. Si se llama a `cuadrado.setAncho(5)`, internamente también debe llamar a `setAlto(5)`, rompiendo el comportamiento esperado de un `Rectangulo` (viola LSP).

Cómo cumplirlo

Una clase abstracta `FormaGeometrica`. Las subclases `Circulo`, `Rectangulo` y `Triangulo` heredan de ella y cada una implementa `calcularArea()` de manera específica para su forma, introduciendo solo las propiedades que necesita.

Sustitución

Los módulos deben poder ser reemplazados por otros compatibles sin que el sistema deje de funcionar correctamente.

Relación Principal: Correspondencia directa y literal con el Principio de Sustitución de Liskov (LSP) de SOLID.

Qué hacer:

  • ✓Asegurar que las clases derivadas puedan ser usadas donde se espere la clase base.
  • ✓Las subclases deben extender la funcionalidad, no restringirla.

Qué no hacer:

  • ✗Lanzar una `NotImplementedException` en un método heredado.
  • ✗Cambiar el comportamiento esperado de un método heredado.

Preguntas que debo hacerme:

  • ¿Puedo poner una instancia de mi subclase en una lista de la clase base y que todo siga funcionando?
  • ¿La subclase cumple con el "contrato" de la clase padre?

Características del software:

  • Confiable: El polimorfismo funciona como se espera, sin necesidad de verificar tipos.
  • Flexible: Permite construir jerarquías de clases robustas y coherentes.

Regla Práctica:

"Si puedes usar 'es un tipo de' para describir la relación, entonces la sustitución debería funcionar."

Ejemplo de Código (Conceptual)

Cómo no cumplirlo

Una clase base `Pato` con un método `volar()`. Una subclase `PatoDeGoma` hereda de `Pato`, pero su método `volar()` lanza una excepción o no hace nada, rompiendo la expectativa de que cualquier `Pato` puede volar.

Cómo cumplirlo

La clase `AnimalQueVuela` tiene un método `volar()`. `Pato` hereda de `AnimalQueVuela`. `PatoDeGoma` no hereda de `AnimalQueVuela` sino de `Juguete`. Si una función necesita un animal volador, recibe un `AnimalQueVuela`.
